MQTT物联网通信

📑 介绍MQTT，以及在TypeScript中使用

简介

Message Queuing Telemetry Transport（消息队列遥测传输）是一种轻量级的物联网消息传输协议，它专为资源受限的设备和不可靠网络设计，核心特点：

  • 基于发布/订阅模式，通信双方解耦，不需要知道对方的存在
  • 协议头部最小仅 2 字节，适合低带宽、高延迟、网络不稳定的环境
  • 支持三种服务质量（QoS）等级，从”尽力而为”到”精确一次送达”
  • 内置心跳保活、遗嘱消息、保留消息等机制，适应设备频繁上下线的场景

拿订阅报纸来类比 MQTT 的工作方式：

  • 发布者：报社，只管出报纸发消息
  • 订阅者：读者，只收取自己关心的栏目
  • 主题：报纸栏目，比如”经济新闻”、”体育新闻”
  • 服务器（Broker）：邮局，负责将报纸投递给订阅了对应栏目的读者

发布者和订阅者不需要知道对方的存在，也不用直接连接，全部通过服务器中转。即使有一方临时断网消息也能暂存后补发

三个核心角色：

角色 说明
发布者（Publisher） 向特定主题发送消息的设备或应用（例如温度传感器上报温度到 /sensor/temp）
订阅者（Subscriber） 订阅一个或多个主题来接收消息的设备或应用（例如手机 App 订阅 /sensor/temp 来展示实时温度）
代理（Broker） 消息的中转站，负责接收发布者的消息并转发给所有订阅了对应主题的客户端（例如 Mosquitto、EMQX）

一个客户端可以同时是发布者和订阅者。比如智能灯控既发布自身状态，也订阅远程开关指令。

与http和websocket对比

使用 mqtt 而不是 http：

场景 MQTT HTTP
卫星信号、弱网环境 ✅ 头部最少 2 字节 ❌ 头部几百字节
设备频繁断网重连 ✅ 内置 Keepalive + 断线续传 ❌ 无状态，需手动重连
一对多消息分发 ✅ 一次发布，多人接收 ❌ 要给 100 台设备发消息，得发 100 次请求
双向实时通信 ✅ 发布/订阅都是实时的 ❌ 只能用轮询（一直问）或 WebSocket

http 是“请你帮我干件事”；mtqq 是“你们关心这件事，我会通知你们”

web socket 也能双向实时通信，为什么还要用 mqtt？

场景 MQTT WebSocket
协议定位 应用层协议（定义消息格式和规则） 传输层通道（只是打通了一条“双向管道”）
消息模式 发布/订阅（一对多，解耦） 点对点（需要自己实现转发逻辑）
服务质量 ✅ QoS 0/1/2（消息不丢、不重） ❌ 无，丢了就是丢了
断线续传 ✅ 持久会话，重连后自动补发 ❌ 重连后啥也没有
消息保留 ✅ Retained Message，新订阅者立即可见 ❌ 需自己实现
协议开销 极小（2 字节固定头） 稍大（至少 2-14 字节帧头）
适用场景 物联网、传感器、推送通知 聊天室、实时游戏、协同编辑

web socket 是一条“高速公路”；mqtt 是高速公路上跑的“邮政系统”——有规则、有服务、能保证送到。

二者一般配合使用

要做什么 选哪个
设备 ↔ 云（弱网、省电） MQTT（原生协议）
浏览器 ↔ 服务器（实时推送） MQTT over WebSocket（浏览器只能走 WS）
实时游戏、聊天室（点对点） 原生 WebSocket（轻量、自定义协议）
传统 API 调用（请求/响应） HTTP（够用，生态成熟）

关键概念

主题（Topic）

主题是消息的地址，用/分层，比如：

可以通配订阅：

  • +/livingroom/temperature：+ 匹配单层
  • /home/#：# 匹配任意多层（# 不能放中间）

比如有主题结构：home/room/device/sensor

服务质量（QoS）

QoS 含义 适用场景
0 最多一次（可能丢） 实时但无所谓丢失的状态数据，如 GPS 位置
1 至少一次（可能重复） 需要收到但可容忍重复，如控制指令
2 恰好一次（确保唯一） 计费、开关等严格不重复的场景

保留消息（Retained Message）

新订阅者上线后，Broker 会把该主题的最后一条保留消息自动发给他。适用于自动获取设备最新状态，比如订阅后知道灯是开着还是关了
