进阶用法

多轮对话

API 是无状态的，服务端不记录上下文，每次请求都需要自己拼接完整的 messages 历史。

关键步骤：每轮对话后，把模型的回复 push 到 messages，再把用户的新问题 push 进去，传给下一次请求。

每次请求时 messages 数组的结构：

标准多轮对话 = 直接 push 上一轮回答。前面重复的内容再次发送，就是命中缓存，价格会比没命中的对内容便宜得多

流式输出

SSE 标准（Server-Sent Events）实现打字机效果：

  • 前端不需要自己处理 TextDecoder、二进制流、chunk解析
  • 实现打字机最简单、最稳定的方案

数据格式：

SSE 不是特殊协议，就是 HTTP 长连接。版本为 HTTP/1.1 或 HTTP/2

如果用到 Nginx，必须额外配置，否则没有打字机效果

Nginx 会有的默认行为：

  • 缓冲数据（buffer），要等到后端传到一定大小才会一次性传给前端
  • 超时自动断开
  • 压缩导致流阻塞，前端收不到实时数据

所以 SSE / 流式接口必须关闭缓冲、禁用压缩、长连接、不超时。

如果实在用不了 SSE，WebSocket 是一个候补方案，只是有些大材小用了
