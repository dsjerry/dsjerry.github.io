数据类型

数值

类型 字节数 范围/说明
TINYINT 1 字节 -128 ~ 127（无符号：0 ~ 255）
SMALLINT 2 字节 -32,768 ~ 32,767（无符号：0 ~ 65,535）
MEDIUMINT 3 字节 -8,388,608 ~ 8,388,607（无符号：0 ~ 16,777,215）
INT 4 字节 -2,147,483,648 ~ 2,147,483,647（无符号：0 ~ 4,294,967,295）
BIGINT 8 字节 -2^63 ~ 2^63-1（无符号：0 ~ 2^64-1）
DECIMAL(M,D) (M+2) 字节（D>0）
(M+1) 字节（D=0）		 精确小数，M 为总位数，D 为小数位数。例：DECIMAL(10,2) 占 12 字节
FLOAT 4 字节 单精度浮点数，约 7 位有效数字
DOUBLE 8 字节 双精度浮点数，约 15 位有效数字
BOOLEAN 1 字节 别名，实质是 TINYINT(1)，MySQL 无独立布尔类型

字符串

类型 最大长度 说明
VARCHAR(N) 65,535 字节（受行大小限制） 可变长度字符串。N 是字符数，实际占用字节数取决于字符集（UTF-8 下每个中文占 3 字节）
CHAR(N) 255 字符 定长字符串，长度固定，检索速度快，适合存储固定长度数据（如手机号、MD5）
TINYTEXT 255 字节 短文本，适合存储简短描述
TEXT 65,535 字节（64 KB） 普通长文本，无法设置默认值，有额外存储开销
MEDIUMTEXT 16,777,215 字节（16 MB） 较长文本，如文章正文
LONGTEXT 4,294,967,295 字节（4 GB） 超大文本
ENUM 1~2 字节 枚举类型，最多 65,535 个值，不建议在业务中频繁变动
SET 1~8 字节 集合类型，最多 64 个值

⚠️ VARCHAR 最大 65,535 字节，不是字符数。UTF-8 字符集下，实际最多约 21,844 个中文字符（65,535 / 3）。

二进制

类型 最大长度 说明
TINYBLOB 255 字节 小型二进制数据
BLOB 65,535 字节（64 KB） 普通二进制数据（图片、文件片段）
MEDIUMBLOB 16,777,215 字节（16 MB） 中等二进制数据
LONGBLOB 4,294,967,295 字节（4 GB） 超大二进制数据（视频、大型文件）
BINARY(N) N 字节 定长二进制字符串
VARBINARY(N) N 字节（最大 65,535） 可变长二进制字符串

💡 BLOB 与 TEXT 的区别：BLOB 是二进制存储（大小写敏感），TEXT 是文本存储（受字符集影响，大小写不敏感）。

日期时间

类型 字节数（无微秒） 带微秒（5.6.4+） 范围
DATE 3 字节 — 1000-01-01 ~ 9999-12-31
TIME 3 字节 3~6 字节 -838:59:59 ~ 838:59:59
DATETIME 8 字节 5~8 字节 1000-01-01 00:00:00 ~ 9999-12-31 23:59:59
TIMESTAMP 4 字节 4~7 字节 1970-01-01 00:00:01 UTC ~ 2038-01-19 03:14:07 UTC
YEAR 1 字节 — 1901 ~ 2155（或 0000）

⚠️ 2038 年问题：TIMESTAMP 最大只能存到 2038 年，业务长期存续建议用 DATETIME。

JSON

类型 最大大小 说明
JSON 1 GB（由 max_allowed_packet 限制） MySQL 8.0 原生 JSON 类型，支持索引（生成列）、JSON 函数查询

使用建议

场景 推荐类型 避免使用
用户 ID、自增主键 BIGINT UNSIGNED 或 INT UNSIGNED 用 VARCHAR 做主键（性能差）
用户名、标题、短文本 VARCHAR(50) ~ VARCHAR(255) 滥用 TEXT（额外开销且无默认值）
文章正文、长描述 TEXT 或 MEDIUMTEXT 用 VARCHAR 超过 64KB（截断风险）
状态标志（0/1） TINYINT(1) 用 BOOLEAN（别名，易误解）
创建时间、更新时间 DATETIME（推荐）
TIMESTAMP（跨时区需求）		 用 INT 存时间戳（可读性差）
价格、金额 DECIMAL(10,2) 用 FLOAT / DOUBLE（精度丢失风险）
配置、扩展字段 JSON 用 TEXT 存序列化字符串（无法高效查询）
身份证、手机号 CHAR(18) / CHAR(11) 用 VARCHAR（定长数据用 CHAR 性能更好）
MD5 / UUID CHAR(32) / CHAR(36) 用 VARCHAR（定长用 CHAR 更优）

设计原则

原则 说明
字段 NOT NULL 尽量不给 NULL，用默认值代替
字段类型合适 存状态用 TINYINT 而非 VARCHAR(255)
不要存 JSON 大字段 除非明确需要，否则拆成关联表
软删除 用 deleted 标记而非物理删除
不要过度设计 先满足需求，再考虑优化

为什么需要事务

sql 层面

ACID 特性

特性 说明 例子
Atomicity（原子性） 要么全做，要么全不做 转账：扣钱和加钱必须同时成功
Consistency（一致性） 数据始终符合约束 金额不能为负数
Isolation（隔离性） 并发事务互不干扰 我转账时你不会看到中间状态
Durability（持久性） 提交后数据不丢失 断电也不会丢

隔离级别

级别 脏读 不可重复读 幻读 性能
READ UNCOMMITTED ✅ ✅ ✅ 最快
READ COMMITTED ❌ ✅ ✅ 较快
REPEATABLE READ（MySQL 默认） ❌ ❌ ✅ 中等
SERIALIZABLE ❌ ❌ ❌ 最慢

索引

为什么需要索引

索引类型

索引设计原则

原则 说明
高频查询字段加索引 WHERE、JOIN、ORDER BY 的字段
不要给所有字段加索引 索引会降低写入速度
联合索引字段顺序 选择性高的放前面
小表不需要索引 几百行数据全表扫描比索引快
避免在索引列上用函数 WHERE DATE(create_time) = '2024-01-01' 用不到索引

EXPLAIN 分析

关键字段解读：

字段 值 含义
type ALL 全表扫描（⚠️ 需要优化）
ref / range 用了索引（✅ 好）
const 唯一索引查找（✅ 最好）
rows 10000 扫描了多少行（越小越好）
key idx_user 实际使用的索引名
Extra Using where 用 WHERE 过滤
Using index 覆盖索引（✅ 最好）
Using filesort 需要额外排序（⚠️ 考虑加索引）

常见问题与解决

现象 原因 解决

写入慢 索引太多 删除不必要的索引
