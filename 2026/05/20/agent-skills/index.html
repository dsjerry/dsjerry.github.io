Agent Skill 快速入门

给 AI Agent 装”领域经验”的标准化打包方式——写一个 SKILL.md 就能让 Agent 学会工作流。

介绍

是什么

Agent Skill 是一个轻量的开放格式，把专业知识和操作流程打包给 AI Agent 用。来自 Anthropic，最初随 Claude Code 发布，现已成为社区标准。

一个 Skill 就是一个文件夹，里面至少有一个 SKILL.md 文件：

它的定位跟 RAG 不一样：RAG 给 Agent 喂知识，Skill 教它流程、踩坑和项目规范。

为什么要用 Skill

Agent 能做的事越来越多，但有两样东西始终缺：领域上下文和踩坑经验。

没有 Skill 的时候，每次都得手写：”用这个库、注意那个坑、输出格式要这样……” 有了 Skill，一次性写好，Agent 每次拿到任务就自动加载。

对比其他方案：

方案 定位 什么时候用
Skill 告诉 Agent 怎么做 需要重复执行的操作流程、领域规范
MCP 告诉 Agent 有什么可用 需要连外部系统（数据库、API）
RAG 告诉 Agent 相关知识 需要查私有文档、知识库
Fine-tuning 改模型本身的行为 需要从根本上提升某方面能力

Skill 和 MCP 是互补的：MCP 管”连什么”（工具接口），Skill 管”连上了怎么用”（操作流程）。

渐进式披露

Agent 加载 Skill 是三阶段的：

  1. 发现阶段 — 启动时只读 name + description，轻量扫描所有 Skill
  2. 激活阶段 — 任务匹配到描述，读取完整的 SKILL.md
  3. 执行阶段 — 按指令操作，需要时调脚本或读参考文件

这意味着可以装几十个 Skill，但上下文开销只取决于当前任务激活的那几个。

SKILL.md 规范速查

Frontmatter 字段

字段 必填 说明
name 是 1~64 字符，小写字母 + 数字 + 连字符
description 是 1~1024 字符，描述功能和触发时机
license 否 许可证
compatibility 否 环境要求（如需要 Python 3.14+）
metadata 否 自定义元数据，如 author, version

name 的坑： 必须跟文件夹名一致。不能有大写、不能以连字符开头结尾、不能有连续连字符。

description 的坑： 太笼统 Agent 不触发。”Helps with PDFs” 没人理。”Extracts text and tables from PDFs, fills forms. Use when working with PDFs” 就对了。

正文

正文就是 Markdown，随便写。建议包含：

  • 分步操作指令
  • 输入输出样例
  • 常见边界情况

文件太大？把参考材料拆到 references/ 目录下，Agent 按需加载。

最佳实践

来自 agentskills.io 官方文档。

从真实经验里提取

不要让 LLM 凭空生成 Skill——它会的不需要记，它不会的才需要记。靠谱的做法：

  1. 跟 Agent 完成一次真实任务，中途纠正它、告诉它偏好
  2. 把”正确路径”、纠正的地方、边界情况提取出来
  3. 打包成 Skill

这样产出的 Skill 里才有真实经验——项目惯例、团队术语、第三方 API 的坑，全在里面。

加踩坑章节，不要只写流程

只有积极指令的 Skill 质量有限。加一个 ## Gotchas 或 ## 踩坑 章节，把反复出问题的地方写进去，效果翻倍。

用模板和清单

  • 输出模板：比如文档生成 Skill 里贴一份 markdown 模板
  • 检查清单：多步工作流用 checklist 防止漏步骤
  • 验证循环：写完检查、部署验证、确认无误再结束

在真实执行中打磨

第一次写的 Skill 大概率不够好。跑几次真实任务，看 Agent 有哪些地方没做好，迭代补上。这不是失败——Skill 本来就是活的。

验证 Skill

官方提供了 skills-ref 验证工具：

它会检查 frontmatter 字段是否符合规范、命名是否合规。

哪些工具支持 Agent Skills

不少 AI 编程工具和 Agent 产品已经支持这个格式。官方列出已有：

  • Junie（JetBrains）
  • Gemini CLI（Google）
  • Claude Code（Anthropic，原创者）
  • OpenCode
  • Cursor
  • OpenHands
  • Windsurf
  • Mux

包括 Windsurf、Cursor 等主流 AI 编程工具都已支持。同一个 Skill 在不同工具之间通用——写一份就行。

什么时候该写 Skill

不是每件事都值得写成 Skill。看几个条件：

操作流程固定、需要重复做？涉及项目独有的规范/约定？踩过坑、希望下次避开？要多步编排（建表→迁移→验证）？——满足两三条就值得写。

反过来，Agent 本身就擅长的事（比如改 CSS 颜色）、一次性任务、不常做的事——需要时再写。
